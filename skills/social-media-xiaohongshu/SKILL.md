# Social Media - Xiaohongshu

这是小红书平台专属 skill。

## 作用范围

- 打开小红书首页 / 帖子页
- 复用登录态
- 浏览帖子与评论区
- 执行评论测试
- 沉淀真实跑通的小红书站内流程

## 当前已验证流程

### 图文发帖链路（2026-03-24 已实战打通）

目标：在小红书网页端完成一次真实图文笔记发布。

#### 已验证前提
- 使用 OpenClaw 管理的浏览器/CDP 环境
- 已完成小红书登录
- 可打开 `https://creator.xiaohongshu.com/publish/publish?source=official`
- 本地已有待上传图片，且位于 `/tmp/openclaw/uploads/` 可访问目录

#### 推荐执行步骤
1. 打开小红书首页，确认顶部出现“我”，说明登录态存在
2. 进入创作中心发布页
3. 若默认落在“上传视频”，切到“上传图文”
4. 查找 `input[type=file]`
5. 将本地图片文件注入该 input（多图上传）
6. 若页面未自动进入编辑态，可再点击一次“上传图片”按钮推进
7. 进入编辑页后填写标题
8. 向正文 `contenteditable` 区域写入正文与话题
9. 点击“发布”
10. 检查页面是否出现“发布成功”提示

#### 本轮实战细节
- 创作中心登录页不一定提供二维码；主站登录弹窗更容易拿到扫码入口
- 图文发布页默认有时会回到“视频上传”，需要主动切到“上传图文”
- 文件上传应使用 `/tmp/openclaw/uploads/...`，避免路径权限问题
- 上传后页面有时不会自动切到编辑态，可补点“上传图片”按钮
- 标题可通过 `input[placeholder*=填写标题]` 写入
- 正文区可通过 `contenteditable` 容器直接写入并触发 `input` 事件
- 成功判据不是按钮变灰，而是页面出现“发布成功”提示
- 若浏览器已由 OpenClaw 管理并占用 profile，不要再试图用 Playwright 新开同一持久化 profile；更稳的做法是继续复用当前 CDP/浏览器会话
- 已补推荐前置脚本：`comic-gen/scripts/prepare_xiaohongshu_publish_payload.py`，用于把标题、正文、图片列表整理成统一发布 payload，供浏览器层直接消费

#### 当前已跑通样例
- 轻漫画图文笔记：1 张封面 + 4 张 panel
- 标题：`我不是陪聊 AI，我是主人的信息哨兵`

### 评论发布链路（2026-03-24 已实战打通）

目标：在小红书网页端完成一次真实评论发布测试。

#### 已验证前提
- 使用 OpenClaw 管理的浏览器/CDP 环境
- 已完成小红书登录
- 可打开帖子详情页并看到评论区

#### 推荐执行步骤
1. 打开小红书首页
2. 若未登录，截图二维码发给用户扫码
3. 用户扫码后，重新检查页面顶部是否出现“我”入口，确认登录态生效
4. 打开目标帖子详情页
5. 滚动/定位到评论区
6. 查找评论编辑区（网页端可能不是标准 textarea，而是 `contenteditable` 容器）
7. 将评论文本写入编辑区
8. 找到“发送”按钮并执行点击
9. 重新抓取评论区，确认新评论已出现

#### 本轮实战细节
- 评论编辑区在网页端实际表现为 `p[contenteditable="true"]`
- 直接使用 DOM 写入 + `InputEvent` 触发，可让“发送”按钮从 disabled 变为可点击
- “发送”按钮可通过页面中包含“发送”文案的 button 定位，不必死依赖固定 aria ref
- 发布后应重新抓取评论区，检查新评论是否真实出现，而不是只看输入框是否清空

#### 本轮测试评论
`测试评论：OpenClaw 链路已打通，留个脚印。`

#### 成功判据
- 评论区出现新的评论记录
- 可见当前账号名
- 时间显示为“刚刚”或对应最新时间

## 设计原则

- 对小红书这类反爬/动态渲染平台，优先走浏览器真实页面路径，不依赖静态 fetch 猜测站内内容。
- 登录态复用比临时 cookie 注入更稳。
- 平台专属流程要记录真实有效的 DOM 特征、按钮状态变化与成功判据。
